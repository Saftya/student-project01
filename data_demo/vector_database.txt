A vector database stores embeddings and makes them searchable by similarity. In a classic database, a query might ask for rows where a column equals a value. In a vector database, the query is itself a vector, and the database returns vectors that are nearby. This is the core infrastructure behind semantic search. A document is split into chunks, each chunk is embedded, and those embeddings are inserted into an index. When a user asks a question, the question is embedded and compared against the stored vectors.

Exact nearest neighbor search can be expensive when there are millions of vectors. Approximate nearest neighbor search, often shortened to ANN, trades a small amount of accuracy for much faster search. FAISS is a popular local library for fast vector search, especially in research and prototyping. ChromaDB is often used in learning projects because it provides persistence, collections, metadata, and a friendly Python API. Both tools help developers avoid writing low-level vector indexing code from scratch.

Many vector databases use graph-based or quantization-based indexing strategies. HNSW, or Hierarchical Navigable Small World graphs, is a common ANN technique. It organizes vectors into a graph that can be searched quickly by moving through likely neighbors. The result is not guaranteed to be perfect, but it is usually good enough for retrieval systems when combined with evaluation. Engineers tune parameters to balance speed, memory, and recall.

Metadata filtering is another important feature. A system may store chunks from many sources, dates, authors, or departments. A user might ask to search only policy documents from 2024 or only engineering notes from a specific project. Metadata filters narrow the candidate set before or during vector search. This is useful because semantic similarity alone does not understand access control, freshness, or business rules. A good retrieval system combines vector similarity with metadata constraints.

The retrieval pipeline is often described as document to chunk to embed to index to query to retrieve. Each step can improve or damage quality. If document loading loses tables, retrieval will never find table facts. If chunks are too small, they may not contain enough context to answer. If chunks are too large, the retrieved context may contain noise. If the embedding model is weak for the domain, similar meanings may not be nearby. If the top_k is too small, the relevant chunk may be missed.

Evaluation makes these tradeoffs visible. Recall@5 asks what percentage of questions have a relevant chunk somewhere in the top five retrieved results. Mean reciprocal rank, or MRR, rewards systems that place the first relevant result near the top. These metrics should be measured before connecting a generator. A RAG system with poor retrieval will hallucinate regardless of how good the language model is, because the generator is being handed the wrong evidence.

Reranking is often added after vector search. The vector database quickly retrieves a candidate set, such as the top ten chunks. A reranker then reads the query and each candidate more carefully and reorders them. This two-stage pattern is common because vector search is fast and broad, while reranking is slower but more precise. The goal is to give the generator a small set of high-quality chunks instead of a noisy pile of loosely related text.

Persistence matters in real applications. If an index disappears every time the program exits, the system must re-embed all documents on every run. ChromaDB can persist a local collection to disk, which is helpful in workshops and prototypes. Production systems may use managed vector databases, access control, monitoring, backup, and data refresh jobs. The core idea remains the same: store embeddings, search by similarity, evaluate retrieval, and only then generate answers.

Vector databases are sometimes treated as a magic layer, but they are only one part of the system. The quality of the source documents, the chunking policy, the embedding model, the query wording, and the evaluation set all shape the result. A small local index can teach the same ideas as a production-scale system because the pipeline is identical. The main difference is scale, operational discipline, and the cost of mistakes.
